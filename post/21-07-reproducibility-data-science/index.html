Reproducibility in Data Science for Beginners

Jul 22, 2021 6 min read

In science, reproducibility matters. My generation of graduate students is being drilled on the importance of reproducibility and responsibility in research due to the “reproducibility crisis” often commented upon in different papers. We are fortunate that an ever-increasing amount of research can be done via computers.

A great resource for learning research computing skills for beginners are the workshops and lessons made available by Software Carpentry.

An Example - NYT COVID Data

Before I share my advice, I want to discuss an example of a well-organized data set: the NYT COVID Dataset. Let’s take a look at this project and why it works. The first thing to note is that it is hosted on GitHub, making it publicly accessible. A great benefit of this is that now anybody can look at the data and confirm or dispute any claims made about what the data say. The format of the project as a whole is that of a git repository. All of the data and what goes with it is hosted in one place and everything is tracked for chances. Git is a version-control system that is easy to use. It was designed for software development to allow users to keep track of changes in the code and giving them the ability to quickly revert files to any previously saved state (think “checkpoints” in other software). The benefit for the non-coder is that it keeps track of who did what in text-based files. This is especially useful for group projects because every line of text in the project can be traced to an author, when it was changed, and what it looked like before that change.

In the case of the NYT data, all data is saved in a text-based format (CSV). If feasible, I always encourage saving data in open and preferably text-based formats. Those types of files will remain accessible for a long time; think about how many files your lab may have that require some old Windows 95 machine somewhere with a licensed product installed. What if the machine stops working, and the manufacturer’s license verification system won’t work for any number of reasons on a VM? With text-based data this won’t happen to you. If you like Excel, using CSV allows interoperability for a larger number of other software packages that may not support reading XLSX files otherwise, while you are still able to use Excel as needed.

The fact that it is a git repository also encourages the creation of a paper trail in the form of the commit messages. Every commit (think checkpoint) is accompanied by a commit message. While in practice these may be short and cryptic, it encourages you to try and think about what you have done and why this checkpoint is here. What happened between this version of the files and the ones before? Writing good commit messages can be invaluable.

One of the key aspects of the NYT repository is the pervasive use of readme files. Every folder includes a README.md file, which is a Markdown formatted text file. Like the CSV these can be read from any system, even from a remote command line based

One Project - One Folder

What if you use the same data set for multiple projects? In many cases, it is entirely feasible to keep a copy of your data in each project folder. Modern-day storage is cheap. Many universities have systems in place for long-term storage of research data (“archiving”) for researchers, for example the mass storage system at UNC. It is entirely reasonable to create a project folder with all the necessary scripts and data sets which can then be compressed into a tar, zip, or 7z archive and then saved as such for long-term storage. If you get an email about your paper in five years time, everything needed to reproduce the data is in one spot.

Creating a Digital Paper Trail

Write scripts. Scripts are great, even and especially for parts you might normally do by hand. Why? Because their existence gives you a paper trail. Have you ever wound up downloading a data set and years later not remembered exactly where you went on what website to find it? Write a shell-script to download it for you instead and you will always have a copy of the link for your records. Even if it’s just a simple curl or wget - saving it as a script provides reproducibility.

If you catch yourself trying to manually copy data presented as a table on a website or typing it off from a PDF report - stop! This is an error prone process and it is not good for reproduciblity. If you use Python, Pandas can scrape tables from a website. The Tabula package can read tables out from a PDF, see this post for an example. While the returned tables may need editing, this too can be done straight from a Python script.

Comment your scripts. The logic of your script often makes sense as you are writing it. But will it make sense to you five months from now? Will it make sense to the next graduate student whose work will be a continuation of yours and who needs to replicate some of your results or steps? Adding comments can save you and others a lot of headache. Every script should have a header that identifies the file, what it does, and who wrote it. If there are options that might need to be changed in subsequent runs of the script, you should explain this at the top of the file.

Datasets should have codebooks. Once you have a dataset, you want sufficient documentation on the dataset to know how to get started with the analysis. In its most basic form you want the codebook to list the variables in your dataset by their variable names (if present in the dataset file itself) in column order together with a label explaining the variable in more detail. In a medical dataset, you might have a variable called weight. This should be given an easy to follow label, e.g. “Patient Weight (kg)”. The variable format should also be included to make writing code that reads the data correctly easier. For example, you might say that weight is given as an int, or if you use decimals it could be indicated by float. If you know what software you will use to read the dataset, you may want to use the specific datatypes from your language, for example 10.2 for a decimal format in SAS or int32 in NumPy. A great codebook is easy to parse by a machine, so use something like a CSV or TAB-delimited file, or even a Markdown table, instead of a Word document or PDF. Some additional tips on writing a good codebook can be found on Lawrence Joseph’s page.

Consider using Jupyter. When used responsibly, Jupyter notebooks are great.
